# Phase 5 ‚Äî UI Design Contract: Tab Management

> Visual and interaction contract for the tab management phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> This is a **WPF desktop application** using WPF-UI 4.2.0 (Fluent Dark). All values are in device-independent pixels. All colours use `DynamicResource` tokens (or the project's custom `Deskbridge*Brush` resources) ‚Äî never hardcoded `#RRGGBB` except where REFERENCE.md ¬ßUI Design ¬ßTheme explicitly names an accent (`#F44747`, `#89D185`, `#FFCC02`).
> Phase 5 **continues** the visual system established in Phases 2 and 3. All spacing, typography, colour, and styling rules carry forward unchanged. Phase 5 adds new visual elements **inside** the existing tab bar DataTemplate and inside the existing `ViewportGrid`; it does NOT modify the outer shell (title bar, icon rail, panel, status bar dimensions, empty-state placeholder).

### Persistent Host Container Layout (inside `ViewportGrid`)

```
Border (Grid.Row=1, Background=ApplicationBackgroundBrush)
 ‚îî Grid x:Name="ViewportGrid"
    ‚îú StackPanel empty-state placeholder  (existing from Phase 2)
    ‚îú Image x:Name="ViewportSnapshot"     (existing from Phase 4 airspace swap)
    ‚îî Grid   x:Name="HostContainer"       (NEW ‚Äî Phase 5, one child per open tab)

        ‚îÇ   ‚îú WindowsFormsHost (from RdpHostControl)
        ‚îÇ   ‚îî ReconnectOverlay (Phase 4) ‚Äî only parented if that tab has ever entered Reconnecting state

```

**Critical invariant (D-04 "never re-parent"):** every per-tab slot is added to `HostContainer` on `HostMounted` and removed only on tab close (`HostUnmounted`). Tab switch flips `Visibility` per slot. No element ever changes parent after first mount until the tab closes. Document this at the top of both `HostContainer`'s XAML comment and `TabHostManager`'s XML doc ‚Äî future refactors must not break it (see WINFORMS-HOST-AIRSPACE.md).

## Per-Tab State Visual Contract (D-12)

Four mutually-exclusive `TabState` values drive the Phase 5 per-tab visuals. This table is the single source of truth for executor and auditor.

| TabState | Visible in the tab title row | Colour | Also visible when tab is inactive? | Source event |
|----------|------------------------------|--------|------------------------------------|--------------|
| `Connecting` | `ui:ProgressRing` 12px spinning, placed left of title | WPF-UI default ring colour (indeterminate, accent) | **Yes** ‚Äî visible on both active and background tabs | Enter: `ConnectionRequestedEvent` ‚Üí pipeline start. Exit: `ConnectionEstablishedEvent`. |
| `Connected` | Title only ‚Äî no dot, no ring | ‚Äî | N/A ‚Äî default empty state | Enter: `ConnectionEstablishedEvent`. Exit: any of the other three. |
| `Reconnecting` | 8px `Ellipse`, Fill `DeskbridgeWarningBrush` (amber `#FFCC02`), left of title | `#FFCC02` | **Yes** ‚Äî background-tab drop visibility is the whole point of D-12 | Enter: `ReconnectOverlayRequested` fired for this connection while a backoff loop is running. Exit: `ConnectionEstablishedEvent` on successful reconnect, or transition to `Error`. |
| `Error` | 8px `Ellipse`, Fill `DeskbridgeErrorBrush` (red `#F44747`), left of title | `#F44747` | **Yes** | Enter: auth-failure or retry-cap-hit from `DisconnectReasonClassifier`. Exit: user actions the reconnect overlay (manual Reconnect) or closes the tab. |

**Mutual exclusion rule:** At most **one** of `ProgressRing` / amber `Ellipse` / red `Ellipse` is visible at any time. The three visuals share the same grid cell ‚Äî the `TabState` enum drives a single `DataTemplateSelector` or a trio of `Visibility` bindings (executor's choice).

**Active-tab accent border does NOT change with `TabState`.** The top 2px accent stays on the active tab in every state (including Error) so users can still identify "which tab am I looking at right now". The state dot communicates health orthogonally.

**Tooltip copy per state** (tooltip on the tab itself, not on the dot):

| State | Tooltip |

## Reconnect Overlay ‚Äî Per-Tab Placement (D-14)

The Phase 4 `ReconnectOverlay` is reused without visual changes. Phase 5 changes only its **parent**:

- Phase 4: overlay parented directly to `ViewportGrid` (single-host assumption).
- Phase 5: overlay parented to the per-tab slot (`HostContainer.Children[tabIndex]`). Its visibility automatically follows its parent's `Visibility` ‚Äî no extra show/hide code on tab switch.

**Background-tab drop UX:**
- `ReconnectOverlayRequested` for an inactive tab ‚Üí `TabHostManager` sets the tab's `TabState` to `Reconnecting` (amber dot appears in that tab's label, visible immediately even while the tab is inactive ‚Äî D-12 requirement) AND parents the overlay inside that tab's slot.
- Because the slot is `Collapsed`, the overlay is invisible until the user switches to the tab.
- **No auto-switch focus** to the dropped tab (D-14).
- Auto-retry continues in background. If reconnect succeeds before the user switches over, the amber dot clears and the overlay is detached (standard Phase 4 behaviour).

**Single overlay per tab at any time** ‚Äî replace-on-drop pattern from Phase 4 `CloseOverlay`. A second `ReconnectOverlayRequested` for the same tab replaces the existing overlay rather than stacking.
