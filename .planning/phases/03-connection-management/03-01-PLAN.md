<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App -> File system | connections.json written to %AppData%, user-controlled directory |
| App -> Windows Credential Manager | Credentials stored/retrieved via OS credential store |
| JSON file -> App | connections.json read from disk, could be tampered with |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | connections.json | mitigate | Atomic write via temp+rename prevents partial writes on crash. JSON deserialization uses System.Text.Json (safe by default, no type name handling). Validate version field on load. |
| T-03-02 | Information Disclosure | connections.json | mitigate | NEVER store passwords in JSON. Only CredentialMode enum value stored. Credentials go exclusively to Windows Credential Manager. XML doc on ConnectionModel.ResolvedPassword (if added) warns against serialization. |
| T-03-03 | Information Disclosure | WindowsCredentialService | accept | Credentials stored in Windows Credential Manager are protected by user's Windows login session. No additional encryption needed at ASVS L1 for desktop app with OS-level credential store. |
| T-03-04 | Tampering | connections.json | accept | File is in %AppData% (user-writable). A user tampering with their own config is an accepted risk for a desktop app. No integrity check needed at ASVS L1. |
| T-03-05 | Denial of Service | JsonConnectionStore.Load | mitigate | Handle corrupt JSON gracefully — catch JsonException, log error, start with empty collection. Show InfoBar to user: "Could not load connections.json. Starting with empty connection list." Verified by Load_CorruptJson_ReturnsEmpty test. |
| T-03-06 | Information Disclosure | Logging | mitigate | Never log credential values. Log credential targets (TERMSRV/hostname) but never NetworkCredential.Password. Use [SensitiveData] XML doc annotations. |
</threat_model>
