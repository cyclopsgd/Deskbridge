<objective>
Fix Credential Guard blocking RDP connections by changing the Windows Credential Manager
target from `TERMSRV/{hostname}` to `DESKBRIDGE/CONN/{connectionId}`.

Purpose: Windows auto-discovers TERMSRV/* entries for RDP CredSSP delegation. On machines
with Credential Guard enabled, this triggers "Windows Defender Credential Guard does not
allow using saved credentials" -- even though Deskbridge injects the password directly via
IMsTscNonScriptable. Changing to a DESKBRIDGE/* prefix prevents Windows from finding the
entry during CredSSP negotiation, eliminating the conflict. The password is still injected
via ClearTextPassword on the ActiveX control.

Output: Updated WindowsCredentialService with new target format, startup migration of
existing entries, and unit tests.
</objective>

1. Change the connection target format from `TERMSRV/{connection.Hostname}` to `DESKBRIDGE/CONN/{connection.Id}`:

2. Change `RdpTargetType` from `CredentialType.Windows` to `CredentialType.Generic`. Rationale: `CredentialType.Windows` (CRED_TYPE_DOMAIN_PASSWORD) was only needed because TERMSRV/* is reserved for that type. `DESKBRIDGE/*` targets use Generic freely -- this is already the pattern for `DESKBRIDGE/GROUP/*`.

3. Remove the `LegacyRdpTargetType` constant and the fallback read in `GetForConnection` (the fallback was for old Generic entries under TERMSRV/* -- migration will handle these).

4. Update `DeleteForConnection` to only remove the single Generic entry (no more dual-type cleanup).

5. Update comments to explain the new target format and WHY TERMSRV was abandoned (Credential Guard conflict).

6. Add a `MigrateFromTermsrv(IConnectionStore connectionStore)` method:
   ```csharp
   /// <summary>
   /// One-time migration: moves credentials from legacy TERMSRV/{hostname} targets
   /// to DESKBRIDGE/CONN/{connectionId} targets. Idempotent -- skips connections
   /// that already have credentials under the new target. Called once at startup.
   /// </summary>
   public void MigrateFromTermsrv(IConnectionStore connectionStore)
   {
       foreach (var connection in connectionStore.GetAll())
       {
           try
           {
               // Skip if new target already has credentials
               var newTarget = $"DESKBRIDGE/CONN/{connection.Id}";
               if (CredentialManager.GetCredentials(newTarget, CredentialType.Generic) is not null)
                   continue;

               // Try to read from old TERMSRV/{hostname} target
               // Try DomainPassword first (canonical for TERMSRV/*), then Generic (legacy)
               var oldTarget = $"TERMSRV/{connection.Hostname}";
               var oldCred = CredentialManager.GetCredentials(oldTarget, CredentialType.Windows)
                          ?? CredentialManager.GetCredentials(oldTarget, CredentialType.Generic);

               if (oldCred is null)
                   continue;

               // Write to new target
               CredentialManager.SaveCredentials(newTarget, oldCred, CredentialType.Generic);
               Log.Information("Migrated credentials for {ConnectionName} from {OldTarget} to {NewTarget}",
                   connection.Name, oldTarget, newTarget);

               // Clean up old entries (both types, swallow failures)
               try { CredentialManager.RemoveCredentials(oldTarget, CredentialType.Windows); } catch { }
               try { CredentialManager.RemoveCredentials(oldTarget, CredentialType.Generic); } catch { }
           }
           catch (Exception ex)
           {
               Log.Warning(ex, "Failed to migrate credentials for connection {ConnectionName} ({ConnectionId})",
                   connection.Name, connection.Id);
               // Continue with next connection -- partial migration is fine
           }
       }
   }
   ```

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-Q01-03 | Denial of Service | Migration failure blocks startup | mitigate | Each connection migrated in its own try/catch -- partial migration is fine, app continues. |
